The Rarity of Rational Relationality

Published on Sat 30 March 2013 under

My bank lets me export transaction data. I use it occasionally to gain some insight into the finances of our household. I tried to automate this recently, with dismal results.

My goal was a process that would

  1. import the data from the bank into a database
  2. automatically categorize transactions according to pre-established criteria
  3. let me manually categorize transactions that don't fit pre-established criteria
  4. provide me with visualizations based on a set of pre-defined queries

Step 1 was easy. I wrote a little script importing an OFX file generated by my bank into a HSQLDB database via JDBC. Before tackling steps 2 and 3, I wanted to come up with a good visualization solution since that's the whole point of it all. I planned to use existing tools rather than code my own (I'm trying to stop solving everything by coding). Here's what I found out:

  • LibreOffice Base plus JDBC plus HSQLDB First there was a HSQLDB version mismatch. When I resolved it Base couldn't create a query (no columns could be extracted from table).
  • Kexi plus JDBC plus HSQLDB Kexi doesn't have a driver for accessing JDBC data sources.
  • Skrooge Skrooge is supposed to really understand OFX but for whatever reason it parsed all transactions as credits.
  • LibreOffice Base plus JDBC plus SQLite I could successfully design a query but running it yielded "SQLite only supports TYPE_FORWARD_ONLY cursors".
  • Kexi plus ODBC plus SQLite
Proudly powered by Pelican, which takes great advantage of Python.
